# T001 — Obfuscated PowerShell Execution (T1059.001)

**Status:** Tuned — all 3 patterns now detected after custom rules deployed.
**MITRE:** T1059.001 (command and Scripting Interpreter: PowerShell)
**Tactic:** Execution
**Date:** 2026-06-21

## Objective

Validate that obfuscated or attacker-style PowerShell content is:

1. captured by Script Block Logging (Event 4104).
2. Shipped from win-ep to Wazuh Manager.
3. Indexed in `wazuh-alerts-*`.
4. Matched by a Wazuh rule with appropriate alert level.

## Pre-conditions

- win-ep agent active (ID 001)
- Sysmon installed with sysmon-modular config
- PowerShell Script Block Logging registry key set (verified in Step 13)
- `agent.conf` includes `Microsoft-Windows-PowerShell/Operational` channel
- No custom tuning rules deployed yet (baseline test)

## Steps to reproduce

Run on win-ep (PowerShell as Admin, new session):

    # 1. Base64-encoded command (classic obfuscation pattern)
    $cmd = "Write-Output 'SOc lab test T001'"
    $bytes = [System.Text.Encoding]::Unicode.GetBytes($cmd)
    $encoded = [Convert]::ToBase64String($bytes)
    powershell.exe -EncodedCommand $encoded

    # 2. Invoke-Expression with concatenated string (IEX pattern)
    $a = "Write-"; $b = "Output"; $c = "'IEX concat test'"
    Invoke-Expression "$a$b $c"

    # 3. Download cradle pattern (no actual download, lab-only)
    $client = "New-Obj" + "ect Net.WebClient"
    Write-Output "Simulated cradle string: $client"

## Expected telemetry

- Event ID 4104 with `scriptBlockText` containing the run content
- Sysmon Event ID 1 (Processcreate) for each `powershell.exe` invocation
- channel: `Microsoft-Windows-PowerShell/Operational`

## Expected alert

Baseline (no tuning):

- Wazuh rule `91802` or `92052` (PowerShell suspicious activity) — level 3-5
- If no alert: gap identified, tuning rule required.

## Result

**Status:** Pass — see tuning details below.

### Telemetry pipeline (end-to-end)

| check | Outcome |
| --- | --- |
| 4104 events recorded on endpoint | 62 events |
| Events visible in Discover (T001 filter) | 2 hits (matching `*T001*` literal) |
| Wazuh alerts generated | 2 (baseline) + 4 (after tuning) |
| Alert level | 4 |
| Triggering rule ID | `91837` |
| Rule description | `Powershell executed "Get-content -Stream or Invoke-Expression". Possible string execution as code` |
After tuning rules deployed, all 3 patterns produce alerts at appropriate levels.

### Coverage by attack pattern

| Pattern | Telemetry captured | Default rule fired | Gap |
| --- | --- | --- | --- |
| EncodedCommand (Base64) | Yes | No | No rule matches `-EncodedCommand`, `FromBase64String`, or decoded content |
| Invoke-Expression concat | Yes | Yes (91837, level 4) | None |
| Net.WebClient cradle string | Yes | No | No rule matches `Net.WebClient`, `DownloadString`, or `New-Object` web cradles |

### conclusion

The detection pipeline is functioning end-to-end: PowerShell Script Block Logging captures all 3 obfuscation patterns and ships them to Wazuh. However, the default Wazuh ruleset only matches the `Invoke-Expression` family, leaving two common attacker techniques uncovered:

1. **Base64 Encodedcommand** (T1027) — most common evasion in real intrusions.
2. **Webclient download cradles** (T1105 staging) — payload retrieval primitive.

custom tuning rules are required to close the gap. See `tuning-rules/0100-windows-powershell.xml` (next step).

## Tuning rule deployed

File: `tuning-rules/9999-windows-powershell.xml`

Three new rules added to close the gaps:

| ID | Level | MITRE | Match (regex) | Description |
| --- | --- | --- | --- | --- |
| 100100 | 10 | T1059.001 / T1027 | `-EncodedCommand` or `-enc <Base64>` | PowerShell launched with Base64-encoded command |
| 100101 | 8 | T1027 / T1059.001 | `FromBase64String` or `[Convert]::ToBase64String` | Script performs Base64 encode/decode |
| 100102 | 12 | T1059.001 / T1105 | `Net.WebClient`, `DownloadString`, `DownloadFile`, `Invoke-WebRequest -OutFile` | Network download cradle |

Deployment workflow:

1. Wrote rule to `/var/ossec/etc/rules/9999-windows-powershell.xml` in the Manager container.
2. Set ownership `wazuh:wazuh` and mode `660`.
3. Restarted Wazuh stack — clean load, no warnings.

### Load-order issue resolved

Initial deployment used filename `0100-windows-powershell.xml` (placed before default `0915-win-powershell_rules.xml` in load order). The custom rule referenced `<if_sid>91802</if_sid>`, but rule `91802` had not loaded yet when parser reached `100100`, producing:

    WARNING: (7617): Signature ID '91802' was not found ... in 'if_sid' of rule '100100'.
    WARNING: (7619): Empty 'if_sid' value. Rule '100100' will be ignored.

Renaming the file to `9999-windows-powershell.xml` forced it to load after the default PowerShell ruleset, resolving the dependency. The rule fired correctly on the next re-test.

## Re-test (after tuning)

Re-ran the same 3 obfuscation patterns plus a `FromBase64String` decoder. Results:

| Pattern | Triggering rule | Level | Status |
| --- | --- | --- | --- |
| `powershell.exe -EncodedCommand ...` | 100100 | 10 | Pass |
| `[Convert]::ToBase64String($bytes)` | 100101 | 8 | Pass |
| `[Convert]::FromBase64String(...)` | 100101 | 8 | Pass |
| `Invoke-Expression "$a$b $c"` | 91837 (default) | 4 | Pass (baseline) |
| `New-Obj + ect Net.WebClient` | 100102 | 12 | Pass |

All 3 custom rules fired at the expected levels within the test window. Default rule 91837 continues to fire for IEX patterns as before, demonstrating no regression.

### Known false positive

Rule 100100 also matched the literal string `# Pattern 1: -EncodedCommand` inside a PowerShell comment line that happened to contain the regex pattern. This is acceptable because:

- The regex is intentionally broad to catch obfuscation in any script-block text.
- Real attacker payloads typically reach the same scriptblock chain.
- A future tuning round can add a negative lookahead for comment markers if noise becomes a problem.
